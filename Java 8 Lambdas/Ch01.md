# 前言与简介

## 背景

Java 8 发布于 2014 年，而本书发布于 2015 年。虽然对于如今的我们来说一切理应如此，不过对于经历过那段时间的人来说，Java 8 的发布还是具有革命性的。

虽然我没经历过，不过像是 Java 8 引入的新特性还有函数式编程这种应该算是特别新潮的东西（虽然 Java 至今都没有完全支持函数式编程），我在学习的路上看到很多图书，他们都会大张旗鼓地宣传 Java 8 的一些特性，如此看来那时的人们应该还是花了好一阵子才接受这些变化（尤其是企业级开发吧）。

我之前经常听人说函数式编程编写的代码难以理解并且难以维护，是防御式编程的天然选择，还有实际大部分人都看太不懂函数式编程的代码。虽然有时我也有这种体会（确实看起来抽象又费劲），不过听到这话时我也怀疑是否有些过于夸张了。现在我倒是觉得这是对函数式编程的偏向正面的评价，不过还好 Java 并未完全支持函数式编程，在当前支持的范围内用好应该还是自己追求的目标。

## 前言

多年以来， 函数式编程被认为是少数人的游戏， 这些人总是强调自己在智力上的优越性，认为函数式编程的智慧不适合推广给普罗大众。 写作此书的目的就是为了挑战这种思想，函数式编程并没有多么了不起， 也绝不是少数人的游戏。

对很多人来说，Java 8 提供的函数式编程元素有限：没有单子，没有语言层面的惰性求值，也没有为不可变性提供额外支持。对实用至上的程序员来说， 这没什么大不了的，我们只想在类库级别抽象，写出简单干净的代码来解决业务问题。这样我们就可以把主要精力放在日常工作上了。

> 能够熟练理解并使用函数式，是单纯靠专门地学习和使用；还是可以依赖上层的学习和经验看懂？
> 最近我也在思考我所追求的“技术自由”究竟有多少程度能够实现？

本书将探讨的主题：

- 如何编写出简单、 干净、 易读的代码——尤其是对于集合的操作？
- 如何简单地使用并行计算提高性能？
- 如何准确地为问题建模，并且开发出更好的领域特定语言？
- 如何写出不易出错，并且更简单的并发代码？
- 如何测试和调试 Lambda 表达式？

将 Lambda 表达式加入 Java，并不只是为了提高开发人员的生产效率，业界也对这一特性**有根本性的需求**。

> 这里大部分都是和 Stream 相关，小部分专门讲解 Lambda 表达式
> 值得注意的是框架源码和架构设计的部分。

## 简介

### Lambda 表达式产生的原因

随着硬件的进步，开发类库的程序员使用 Java 时，发现抽象级别还不够。**为了编写这类处理批量数据的并行类库， 需要在语言层面上修改现有的 Java： 增加 Lambda 表达式。**这样做是有代价的，程序员必须学习如何编写和阅读使用 Lambda 表达式的代码，与手写一大段复杂、线程安全的代码相比，学习一点新语法和一些新习惯容易很多。

对于习惯了面向对象编程的开发者来说，抽象的概念并不陌生。**面向对象编程是对数据进行抽象，而函数式编程是对行为进行抽象**，现实世界中，数据和行为并存。另外也不是所有人都在编写性能优先的代码，易读的代码也易于维护、更可靠、更不容易出错，更多地表达了业务逻辑的意图， 而不是它的实现机制。

> Lambda 表达式/函数式编程和并行之间有什么必然关系吗？见：[1.1 Lambda 表达式/函数式编程和并行之间有什么必然关系吗？](Ch01_简介.md#11-lambda-表达式函数式编程和并行之间有什么必然关系吗)

在写回调函数和事件处理程序时，程序员不必再纠缠于匿名内部类的冗繁和可读性，函数式编程让事件处理系统变得更加简单。能将函数方便地传递也让编写惰性代码变得容易，惰性代码在真正需要时才初始化变量的值。

> 这一点其他书有提到过：常规类-匿名内部类类-匿名函数（行为参数化）

### 函数式编程

**函数式编程核心是：在思考问题时，使用不可变值和函数，函数对一个值进行处理，映射成另一个值。**

不同的语言社区往往对各自语言中的特性孤芳自赏。我们关心的是如何写出好代码，而不是符合函数式编程风格的代码。

> 作为一本专业编程书的作者，说出这种观点倒是很奇怪，好代码和好风格我认为是可以共存的。
